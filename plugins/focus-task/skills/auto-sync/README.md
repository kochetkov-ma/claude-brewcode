## Обзор

Auto-sync решает проблему устаревания документации. Скиллы, агенты, правила и markdown-документы со временем расходятся с реальным состоянием кодовой базы. Auto-sync:

1. **Обнаруживает** все `.md` файлы в scope и помечает их тегом `auto-sync: enabled`
2. **Отслеживает** состояние каждого файла через INDEX.jsonl (mtime, hash, версия)
3. **Определяет** устаревшие документы (порог — 7 дней)
4. **Обновляет** содержимое параллельными агентами, сверяя с кодовой базой, локальными документами и веб-источниками

### Вызов

```
/focus-task:auto-sync                        # PROJECT — синхронизация .claude/**
/focus-task:auto-sync global                 # GLOBAL — синхронизация ~/.claude/**
/focus-task:auto-sync status                 # Статус всех документов
/focus-task:auto-sync init ./docs/api.md     # Добавить файл в auto-sync
/focus-task:auto-sync init ./docs/api.md "source: https://api.example.com, focus: endpoints"
/focus-task:auto-sync path/to/file.md        # Синхронизировать один файл
/focus-task:auto-sync ./docs/               # Синхронизировать папку
```

## Режимы работы

Режим определяется аргументами при вызове скилла. Скрипт `detect-mode.sh` парсит аргументы и возвращает формат `MODE|ARG`.

| Режим | Аргумент | Scope | Действие |
|-------|----------|-------|----------|
| `STATUS` | `status` | Все scopes | Диагностический отчёт о состоянии INDEX |
| `INIT` | `init <path> [prompt]` | Один файл | Добавить auto-sync тег + кастомный протокол |
| `PROJECT` | _(пусто)_ | `.claude/**` | Синхронизация проектных документов |
| `GLOBAL` | `global` | `~/.claude/**` | Синхронизация глобальных документов |
| `FILE` | `path/to/file.md` | Один файл | Синхронизация конкретного файла |
| `FOLDER` | `path/to/folder/` | Все `.md` в папке | Синхронизация всех документов в папке |

### Особенности определения режима

- Аргументы нормализуются в lowercase перед сравнением
- Путь с расширением `.md` → режим `FILE`
- Существующая директория или путь с `/` без `.md` → режим `FOLDER`
- Всё остальное → `PROJECT` с аргументами как контекст

## Flow выполнения

### Общая схема

```
Вызов /focus-task:auto-sync [args]
       │
       ▼
┌─────────────────────┐
│  Mode Detection     │  detect-mode.sh
└─────────┬───────────┘
          │
    ┌─────┼──────────────┐
    │     │              │
    ▼     ▼              ▼
STATUS  INIT    Sync modes (PROJECT/GLOBAL/FILE/FOLDER)
  │       │              │
  ▼       ▼              ▼
Report  Tag+Exit  ┌─────────────────────┐
                  │  Phase 1: Setup     │  Инициализация INDEX.jsonl
                  └─────────┬───────────┘
                            │
                            ▼
                  ┌─────────────────────┐
                  │  Phase 2: Discovery │  Поиск всех .md → Auto-Tag
                  └─────────┬───────────┘
                            │
                            ▼
                  ┌─────────────────────┐
                  │  Phase 3: Queue     │  Приоритизация файлов для синхронизации
                  └─────────┬───────────┘
                            │
                            ▼
                  ┌─────────────────────┐
                  │  Phase 4: Process   │  ft-auto-sync-processor (max 5 параллельно)
                  └─────────┬───────────┘
                            │
                            ▼
                  ┌─────────────────────┐
                  │  Phase 5: Report    │  Обновление INDEX, отчёт в reports/
                  └─────────────────────┘
```

### Phase 1: Setup

Инициализация рабочего окружения:

1. Определение `FT_PLUGIN` — пути к плагину в кэше
2. Определение scope (`project` → `.claude/auto-sync/`, `global` → `~/.claude/auto-sync/`)
3. Создание директории INDEX, если не существует
4. Чтение текущего INDEX (если есть)

### STATUS Mode

Диагностический отчёт о состоянии INDEX и отслеживаемых документов.

| Секция | Содержание |
|--------|-----------|
| Scope | Режим, путь scope, области поиска |
| Indexed | Путь, тип, статус, версия, протокол, последняя синхронизация |
| Non-Indexed | Путь, определённый тип, наличие auto-sync тега |
| Summary | Счётчики: indexed (ok/stale/error), non-indexed, total |

**Отдельный flow — не переходит к фазам синхронизации.**

### INIT Mode

Добавление auto-sync тега и кастомного протокола к существующему документу.

**Вызов:** `init <path> [custom protocol prompt]`

| Шаг | Действие |
|-----|----------|
| 1 | Проверка существования файла |
| 2 | Проверка наличия тега (если есть → "Already tagged", EXIT) |
| 3 | Определение типа документа |
| 4 | Добавление auto-sync тега (YAML frontmatter или HTML-комментарий) |
| 5 | Если указан custom protocol → генерация `<auto-sync-protocol>` блока |
| 6 | LLM-оптимизация блока протокола (таблицы, компактный формат) |
| 7 | Добавление записи в INDEX.jsonl |

**Отдельный flow — не переходит к фазам синхронизации.**

### Phase 2: Discovery & Auto-Tag

Обнаружение и разметка файлов:

1. Поиск всех `.md` файлов в scope через `find`
2. Для каждого файла проверка наличия тега `auto-sync: enabled` / `auto-sync:enabled`
3. Если тега нет — автоматическое добавление:
   - **SKILL.md / agents/*.md** → YAML frontmatter `auto-sync: enabled`
   - **Остальные .md** → HTML-комментарий `<!-- auto-sync:enabled -->` после заголовка
4. Слияние с INDEX: новые записи добавляются, удалённые файлы помечаются

### Phase 3: Build Queue

Построение очереди синхронизации с приоритизацией:

| Приоритет | Критерий |
|-----------|----------|
| 1 (высший) | Новый файл (нет в INDEX) |
| 2 | Изменился mtime |
| 3 | Изменился hash (SHA256[:8]) |
| 4 | Устарел > 7 дней |
| 5 | Статус `error` |

**Пропускаются:** файлы с неизменёнными mtime+hash, не устаревшие; файлы с `auto-sync: disabled`.

### Phase 4: Parallel Processing

Обработка очереди агентами `ft-auto-sync-processor`:

- Максимум **5 агентов параллельно** в одном batch
- Каждый агент получает путь, тип и протокол документа
- При >5 файлах — обработка батчами с ожиданием между ними
- Агент возвращает JSON с результатом обработки

### Phase 5: Update INDEX & Report

1. Обновление INDEX.jsonl по результатам агентов (новый hash, mtime, version, status, дата синхронизации)
2. Генерация отчёта в `.claude/tasks/reports/{TS}_auto-sync/report.md`
3. Вызов `ft-coordinator` для финализации

## INDEX.jsonl — формат и управление

### Формат записи

```jsonl
{"p":"skills/auth/SKILL.md","t":"skill","m":1706745600,"h":"a1b2c3d4","pr":"default","v":"1.0.0","u":"2026-02-05T14:00Z","s":"ok"}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `p` | string | Относительный путь от корня проекта |
| `t` | enum | Тип: `skill` / `agent` / `doc` / `rule` |
| `m` | int | mtime файла (Unix epoch, секунды) |
| `h` | string | SHA256[:8] — первые 8 символов хэша содержимого |
| `pr` | string | Протокол: `default` или имя кастомного |
| `v` | string | Семантическая версия документа |
| `u` | string | Дата последней синхронизации (ISO8601) |
| `s` | enum | Статус: `ok` / `stale` / `error` |

### Расположение

| Scope | Путь |
|-------|------|
| Project | `.claude/auto-sync/INDEX.jsonl` |
| Global | `~/.claude/auto-sync/INDEX.jsonl` |

### Статусы

| Статус | Значение | Действие |
|--------|----------|----------|
| `ok` | Синхронизирован в пределах порога | Пропустить |
| `stale` | Старше порога (по умолчанию 7 дней) | Добавить в очередь |
| `error` | Ошибка при последней синхронизации | Добавить в очередь (приоритет 5) |

### Комментарии

Строки, начинающиеся с `#`, игнорируются. Используются для документации формата в шаблоне.

## Типы документов

Тип определяется по frontmatter и пути файла. Первое совпадение побеждает (приоритет по порядку).

| Тип | Критерий обнаружения | Приоритет |
|-----|---------------------|-----------|
| `skill` | `name: focus-task:*` или `name: <project>:*` в frontmatter | 1 |
| `agent` | `model:` **И** `tools:` в frontmatter | 2 |
| `rule` | Путь содержит `/rules/` | 3 |
| `doc` | По умолчанию (любой `.md` с тегом auto-sync) | 4 |

### Regex-паттерны

- **Skill:** `^name:\s*(focus-task|[a-z-]+):[a-z-]+`
- **Agent:** наличие обоих полей `model:` и `tools:` в YAML frontmatter

### Интервалы проверки по типам

| Тип | Интервал по умолчанию |
|-----|----------------------|
| `skill` | 7 дней |
| `agent` | 14 дней |
| `rule` | 3 дня |
| `doc` | 7 дней |

## Протоколы обновления

Протокол определяет **откуда** брать обновления, **как** разбивать документ на блоки для исследования, и **какие правила** применять при обновлении.

### Default Protocol: Skill

**Источники (по приоритету):**
1. Официальная документация Claude Code
2. GitHub issues (anthropics/claude-code)
3. Changelog / release notes
4. Связанные скиллы в том же плагине

**Триггеры обновления:**
- Изменение синтаксиса в документации
- Breaking changes
- Deprecation
- Исправление бага в Claude

**Правила:** сохранять структуру, обновлять только факты (пути, имена, значения), не трогать EXECUTE-блоки, не менять тон инструкций.

### Default Protocol: Agent

**Источники:**
1. Model changelog (обновления Anthropic)
2. Документация инструментов (MCP)
3. Best practices
4. Данные о производительности (токены, качество)

**Триггеры:** новая модель, изменение API инструмента, изменение лимитов контекста.

**Правила:** сохранять идентичность агента, обновлять workflow если паттерны изменились, синхронизировать ответственности с реальным использованием.

### Default Protocol: Doc

**Источники:**
1. URL-ссылки в теле документа
2. Связанные документы в той же директории
3. Web-поиск по теме + год
4. Реализация в кодовой базе

**Триггеры:** 404 на исходном URL, расхождение контента >30%, новая версия, изменение API.

**Правила:** только факты (без изменения мнений), сохранять форматирование, помечать устаревшие секции `<!-- STALE: reason -->`, не удалять контент — только обновлять или помечать.

### Default Protocol: Rule

**Источники:**
1. KNOWLEDGE.jsonl (накопленные знания проекта)
2. Статический анализ кода
3. Паттерны из тестовых падений
4. Командные конвенции

**Триггеры:** новая запись в KNOWLEDGE, нарушение правила >3 раз, правило ни разу не сработало, конфликт правил.

### Custom Protocol

Документ может определить собственный протокол через блок `<auto-sync-protocol>`:

```markdown
<auto-sync-protocol>
sources:
  files: src/**/*.ts
  urls: https://api.example.com/docs
  related: .claude/skills/*.md

split: 3 blocks by section

research:
  block-1: API endpoints focus
  block-2: Config updates
  block-3: Error handling

merge:
  dedupe: true
  priority: code > web

rules:
  - preserve examples
  - update tables only values
  - mark deprecated with <!-- DEPRECATED -->
</auto-sync-protocol>
```

Или через frontmatter: `auto-sync-protocol: <name>` → загружается из `references/protocol-<name>.md`.

## Auto-Tag — автоматическая разметка

При запуске в режимах PROJECT, GLOBAL, FOLDER скилл автоматически размечает все найденные `.md` файлы.

### Логика разметки

```
Для каждого .md файла в scope:
  1. Уже есть auto-sync тег? → пропустить
  2. Есть YAML frontmatter (---)?
     → Добавить строку `auto-sync: enabled` в frontmatter
  3. Нет frontmatter?
     → Добавить `<!-- auto-sync:enabled -->` после первого `# Title`
```

### Форматы тегов

**YAML frontmatter (для skills, agents):**
```yaml
---
name: focus-task:auth
description: ...
auto-sync: enabled
auto-sync-version: 1.0.0
auto-sync-protocol: default
---
```

**HTML-комментарий (для rules, docs):**
```markdown
# My Document
<!-- auto-sync:enabled -->
```

### Полный формат frontmatter

| Поле | Обязательно | По умолчанию | Описание |
|------|-------------|--------------|----------|
| `auto-sync` | Да | — | Должно быть `enabled` |
| `auto-sync-version` | Нет | `1.0.0` | Семантическая версия |
| `auto-sync-protocol` | Нет | `default` | Имя протокола |
| `auto-sync-sources` | Нет | `[]` | Массив URL или поисковых запросов |
| `auto-sync-interval` | Нет | `7d` | Интервал проверки (1d, 7d, 30d) |

## Агент ft-auto-sync-processor

Специализированный агент для обработки **одного** документа. Модель: `sonnet`. Инструменты: Read, Write, Edit, Glob, Grep, Task, WebFetch.

### Workflow агента (7 шагов)

```
Входные данные: path, content, type, protocol
       │
       ▼
┌──────────────────────────┐
│ Step 1: Detect Type      │  Анализ frontmatter и пути
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Step 2: Parse Protocol   │  <auto-sync-protocol> или default
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Step 3: Context Gather   │  3 параллельных Explore-агента (зависят от типа)
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Step 4: Split & Research │  Разбиение на 3 блока по ## заголовкам
│  (3 parallel Explore)    │  Параллельное исследование каждого блока
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Step 5: Aggregate        │  Слияние, дедупликация, приоритизация
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Step 6: Update Document  │  Применение изменений bottom-up (Edit)
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Step 7: Generate Output  │  JSON с результатом
└──────────────────────────┘
```

### Контекстные агенты по типам (Step 3)

**Skill:**

| # | Фокус |
|---|-------|
| 1 | Проверить существование файлов из инструкций (glob) |
| 2 | Проверить соответствие инструкций паттернам кодовой базы |
| 3 | Найти устаревшие примеры (сниппеты vs реальные файлы) |

**Agent:**

| # | Фокус |
|---|-------|
| 1 | Проверить корректность списка инструментов |
| 2 | Проверить соответствие workflow текущим хукам/скиллам |
| 3 | Найти недостающие ответственности (сравнение с аналогичными агентами) |

**Doc/Rule:**

| # | Фокус |
|---|-------|
| 1 | Проверить существование ссылок на файлы |
| 2 | Проверить валидность URL (WebFetch) |
| 3 | Найти связанные изменения в git-истории |

### Приоритеты источников при агрегации

| Приоритет | Источник | Вес |
|-----------|----------|-----|
| 1 | Кодовая база (Grep/Glob) | Высший |
| 2 | Локальные документы | Средний |
| 3 | Web (WebFetch) | Низший |

**Дедупликация:** одна и та же строка — побеждает источник с высшим приоритетом. Конфликтующие исправления — предпочтение evidence из кодовой базы.

### Формат выходных данных

```json
{
  "path": "/path/to/doc.md",
  "type": "skill",
  "version": {"old": "1.0.0", "new": "1.0.1"},
  "changes": [
    {"line": 42, "type": "update", "desc": "Fixed file path reference"},
    {"line": 28, "type": "delete", "desc": "Removed deprecated flag"},
    {"line": 15, "type": "add", "desc": "Added missing tool"}
  ],
  "errors": []
}
```

### Важные правила агента

- Изменения применяются **bottom-up** (по убыванию номеров строк) для сохранения корректности номеров
- Используется `Edit` (не `Write`) для обновлений
- Секции `<!-- user -->` сохраняются без изменений
- EXECUTE-блоки в скиллах не модифицируются
- Версия инкрементируется только при фактических изменениях контента

## Скрипты

### detect-mode.sh

Определяет режим работы из аргументов командной строки.

```bash
bash detect-mode.sh "status"                          # → STATUS|
bash detect-mode.sh "init ./docs/api.md custom prompt" # → INIT|./docs/api.md custom prompt
bash detect-mode.sh "global"               # → GLOBAL|
bash detect-mode.sh ""                     # → PROJECT|
bash detect-mode.sh "path/to/file.md"      # → FILE|path/to/file.md
bash detect-mode.sh "./docs/"              # → FOLDER|./docs/
```

**Выход:** `MODE|ARG` — режим и аргумент через `|`.

### discover.sh

Находит файлы с тегом auto-sync.

```bash
bash discover.sh .claude              # Пути (по умолчанию)
bash discover.sh .claude json         # JSON-массив
```

Ищет два паттерна через `grep -rl`:
- YAML: `^auto-sync:\s*enabled`
- HTML: `<!-- auto-sync:enabled`

### index-ops.sh

CRUD-операции над INDEX.jsonl. Требует `jq`.

| Команда | Использование | Описание |
|---------|---------------|----------|
| `read` | `index-ops.sh read <path>` | Прочитать и валидировать INDEX |
| `add` | `index-ops.sh add <path> '<json>'` | Добавить запись (ошибка если существует) |
| `update` | `index-ops.sh update <path> <doc_path> <field> <value>` | Обновить поле записи |
| `remove` | `index-ops.sh remove <path> <doc_path>` | Удалить запись |
| `query` | `index-ops.sh query <path> <field> <value>` | Найти записи по значению поля |
| `check-stale` | `index-ops.sh check-stale <path> [days]` | Найти устаревшие (по умолчанию 7 дней) |
| `hash` | `index-ops.sh hash <file>` | SHA256[:8] файла |
| `mtime` | `index-ops.sh mtime <file>` | Unix epoch mtime файла |

**Кроссплатформенность:** `shasum -a 256` (macOS) / `sha256sum` (Linux), `stat -f %m` (macOS) / `stat -c %Y` (Linux).

## Обработка ошибок

| Ошибка | Действие | Продолжение |
|--------|----------|-------------|
| INDEX повреждён | Перестроить через discovery | Да |
| Ошибка парсинга протокола | Логировать, использовать default | Да |
| Файл не найден | Пропустить, добавить в errors | Да |
| Таймаут агента | Повторить один раз (60s timeout) | Да (пропустить при повторном сбое) |
| WebFetch ошибка | Добавить в errors, пометить URL | Да |
| Конфликт при Edit | Откатить обновления, добавить в errors | Нет |
| Ошибка Write | Откатить, полный отчёт об ошибке | Нет |
| Нет файлов с тегом | Отчёт "0 found" | Да |
| Вызов `/focus-task:doc` | Deprecation notice — использовать auto-sync | — |

## Версионирование

Автоматическое семантическое версионирование документов.

| Тип изменения | Increment | Пример |
|---------------|-----------|--------|
| Факты, значения, опечатки | Patch (`0.0.x`) | `1.0.0` → `1.0.1` |
| Новые секции, примеры | Minor (`0.x.0`) | `1.0.1` → `1.1.0` |
| Структурные изменения | Major (`x.0.0`) | `1.1.0` → `2.0.0` |

Версия инкрементируется только при фактическом изменении содержимого. Если агент определил, что обновлений нет — bump не происходит.

## Интеграция с focus-task

- **ft-coordinator:** вызывается после завершения для записи отчёта и обновления статуса задачи
- **Протокол:** WRITE report → CALL ft-coordinator (обязательный 2-step протокол)
- **Хуки:** auto-sync не имеет собственных хуков, но работает в контексте focus-task хуков (pre-task инжектирует KNOWLEDGE, post-task следит за протоколом)
