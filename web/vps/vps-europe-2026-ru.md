## NVMe vs SSD — когда имеет значение

| Нагрузка | SSD достаточно | NVMe нужен |
|----------|:--------------:|:----------:|
| Caddy (проксирование, статика) | да | нет |
| Redis (данные в памяти, disk только для snapshot) | да | нет |
| PostgreSQL — простые CRUD, типичный web-запрос | да | нет |
| PostgreSQL — сложные joins, аналитика, высокий RPS | на грани | да |
| CI runner (npm/gradle/maven, сборка Docker-образов) | медленно | да, ощутимо |
| Docker pull / unpack образов | приемлемо | быстрее, но разово |

**Практический вывод для старта:**
- Нет тяжёлых CI или сложных PostgreSQL-запросов → **200 GB SSD** (больше места, важнее на старте)
- CI — приоритет → **100 GB NVMe** или доплатить €2.55/мес за 200 GB NVMe
- Оговорка по Contabo: их NVMe показал 21,400 IOPS (EXPERTE.com) — это слабо для NVMe. Разрыв между NVMe и SSD у них меньше, чем у других провайдеров.

## Стратегия перехода

**Фаза 1 — старт (1-2 месяца): Contabo Cloud VPS 20, €7/мес**
- Без обязательств, можно уйти в любой момент
- Поднять весь стек в Docker Compose: Caddy + PostgreSQL + Redis + приложение
- Цель: убедиться в рабочей конфигурации, оценить реальное потребление ресурсов

**Фаза 2 — оценка (месяц 2-3): решение по ресурсам**
- Если 12 GB RAM / 6 vCPU хватает → остаться на VPS 20
- Если нужно больше RAM или CPU → Contabo VPS 30 (~€11.90/мес annual), без смены провайдера
- Переезд: `docker compose down` на старом → `docker compose up` на новом, данные через pg_dump / volume backup

**Фаза 3 — production (месяц 3+): Netcup RS 2000 G12, €16.89/мес**
- Переход когда: CI-сборки нестабильны, PostgreSQL-запросы плавают, нужна предсказуемость
- 8 dedicated AMD EPYC 9645 Zen 5 + 512 GB NVMe — фиксированная производительность
- Тот же Docker Compose, переезд за вечер
- Обязательство 12 месяцев — брать когда стек стабилен и понятен объём
