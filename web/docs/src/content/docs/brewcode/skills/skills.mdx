# Skills

<Badge variant="primary" text="opus" /> <Badge variant="secondary" text="session" />

Skill management orchestrator.
List all available skills, improve existing ones with activation optimization, or create new skills from a prompt or spec file. Full guided workflow with discovery, user interview, review, and E2E testing.

<CardGrid>
  <Card title="LIST" icon="search">
    Show all skills: global, project, plugin. Runs `list-skills.sh` automatically and presents a summary table.
  </Card>
  <Card title="UP" icon="wrench">
    Improve existing skill: optimize description, fix activation triggers, add references, enhance scripts.
  </Card>
  <Card title="CREATE" icon="pencil">
    Create new skill from a text prompt or spec file. Full guided workflow with testing and review.
  </Card>
</CardGrid>

<Callout type="note" title="Related: Skill Creator Agent">
  This skill orchestrates the <a href="/brewcode/agents/skill-creator/">Skill Creator</a> agent. The skill handles discovery, user interaction, review, and E2E testing. The agent handles creation, validation, unit tests, and README generation.
</Callout>

</Callout>

| Field | Value |

| Command | `/brewcode:skills` |
| Arguments | `[list\|up\|create] [target] \| <skill-path>` |
| Model | opus |
| Context | session |
| Tools | Read, Write, Edit, Glob, Grep, Bash, Task, WebSearch, WebFetch, AskUserQuestion |

## Quick Start

```bash
# List all skills
/brewcode:skills

# Improve a skill by name (shorthand -- "up" is default)
/brewcode:skills commit

# Improve all skills in a folder
/brewcode:skills up ~/.claude/skills/

# Create from a text prompt
/brewcode:skills create "semantic code search"

# Create from a spec file
/brewcode:skills create ./spec.md
```

## Modes

<Tabs>
  <TabItem label="list">
    Lists all skills across three locations: global (`~/.claude/skills/`), project (`.claude/skills/`), and plugins. Runs `list-skills.sh` and outputs a summary table with skill names, locations, and counts.

    **Output format:**

    | Location | Count | Skills |
    |----------|-------|--------|
    | Global (`~/.claude/skills/`) | N | skill1, skill2 |
    | Project (`.claude/skills/`) | N | skill3 |
    | Plugins | N | plugin:skill1 |
  </TabItem>
  <TabItem label="up">
    Improves an existing skill. Accepts a skill name, path, or folder as target.

    **Target resolution:**

    | Target type | Example | Resolution |
    |-------------|---------|------------|
    | Skill name | `commit` | Search in global/project/plugin skills |
    | File path | `~/.claude/skills/commit/SKILL.md` | Use directly |
    | Folder | `~/.claude/skills/` | Find all `*/SKILL.md` in folder -- process each |

    If the first argument is not a mode keyword, it is treated as a target for `up` mode (shorthand).
  </TabItem>
  <TabItem label="create">
    Creates a new skill from a text prompt or spec file.

    | Input | Action |
    |-------|--------|
    | Path to `.md` file | Read as spec |
    | Text prompt | Use as research query |

    Supports **Mode Switcher** pattern detection: if the prompt contains keywords like "mode", "toggle", "switch", or "persistent", the skill asks whether to generate a Mode Switcher skill with `on`/`off`/`status` arguments and state persistence via `brewcode.state.json`.
  </TabItem>
</Tabs>

## Workflow (create / up)

Both `create` and `up` follow the same 7-phase pipeline. Differences are noted per phase.

<Steps>
  <li>
    <div>
      <strong>Phase 0 -- Discovery</strong>
      <p>2-3 Explore agents analyze skill patterns and project structure in parallel. For <code>create</code>: scan existing skill patterns, project structure, and optionally web research. For <code>up</code>: analyze the target skill quality and compare against plugin patterns.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Phase 1 -- User Interview</strong>
      <p>Interactive via <code>AskUserQuestion</code>. Seven steps: check conversation history (create only), determine input type (create only), invocation type (user/LLM/both), Mode Switcher detection (create only), testing depth (Quick/Standard/Deep), review type (Simple/Quorum), and plan confirmation.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Phase 2 -- Create / Improve</strong>
      <p>Spawn <a href="/brewcode/agents/skill-creator/">Skill Creator</a> agent with discovery results and user choices. The agent builds structure, frontmatter, references, and scripts. For folder targets, parallel agents process each skill in one message.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Phase 3 -- Validate</strong>
      <p>Automatic: structure checks, <code>validate-skill.sh</code>, unit tests for scripts, README generation. Runs inside the Skill Creator agent.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Phase 4 -- Review</strong>
      <p><strong>Skip if Quick testing.</strong> Simple review: 1 reviewer + 1 verification agent. Quorum review: 3 reviewers in parallel, 2/3 threshold, then DoubleCheck verification. Confirmed findings are fixed by the Skill Creator.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Phase 5 -- E2E Testing</strong>
      <p><strong>Only for Deep testing depth.</strong> Creates test scenarios (1 per mode + edge cases), executes each in a temp project via <code>claude -p</code>. Failed scenarios retry up to 2 times; major issues loop back to Phase 2.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Phase 6 -- Summary</strong>
      <p>Structured report: action performed, skill path, invocation type, testing depth, review type, phases completed, problems found and fixed, test results, and suggestions.</p>
    </div>
  </li>
</Steps>

## User Interview Details

Phase 1 collects six configuration choices through interactive questions:

| Step | Question | Options |
|------|----------|---------|
| Invocation type | Who will invoke this skill? | User only (slash command), LLM auto-detect, Both (default) |
| Mode Switcher | Create as Mode Switcher skill? | Yes / No (create only, auto-detected from keywords) |
| Testing depth | How thoroughly should the skill be tested? | Quick (default), Standard, Deep |
| Review type | What review approach? | Simple (1 reviewer), Quorum (3 reviewers, 2/3 threshold) |
| Plan confirmation | Proceed with this plan? | Proceed, Adjust, Cancel |

<Callout type="tip">
  Testing depth determines which phases run: **Quick** skips review and E2E. **Standard** adds review. **Deep** adds both quorum review and E2E testing.
</Callout>

## Mode Switcher

<Callout type="note" title="Mode Switcher Pattern">
  The `create` mode can generate **Mode Switcher** skills -- skills that toggle persistent session-level behavioral modes. The generated skill supports three arguments: `on [mode-name]`, `off`, and `status`. State is stored in `brewcode.state.json` with scope priority: session > project > global. Hooks inject mode instructions on every event.
</Callout>

## Review Types

<Tabs>
  <TabItem label="Simple">
    1. Spawn one reviewer (opus) -- review skill quality at target path
    2. If findings: spawn verification agent (sonnet) -- verify against actual code
    3. Confirmed findings: spawn Skill Creator (opus) -- fix verified issues

    Default for **Standard** testing depth.
  </TabItem>
  <TabItem label="Quorum">
    1. Three reviewers in parallel (one message) -- independent skill quality reviews
    2. Quorum threshold: same file + same area (+-5 lines) + same category = 2/3 agree
    3. DoubleCheck: one reviewer (opus) verifies quorum findings against code
    4. Confirmed findings: Skill Creator fixes verified issues

    Default for **Deep** testing depth.
  </TabItem>
</Tabs>
