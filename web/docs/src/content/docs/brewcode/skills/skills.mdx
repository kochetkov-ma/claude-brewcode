# Your skill library, under control.

<Callout type="caution">
**Skills multiply fast.** Every plugin adds more recipes (скиллы), and Claude's auto-detection starts misfiring — wrong skill fires, redundant triggers compete. Maintaining them manually means hand-editing SKILL.md files and hoping the activation description is optimized.
</Callout>

<Callout type="tip">
**One skill, three operations.** `/brewcode:skills` lists what you have, improves what's broken, and creates what's missing — through a guided 7-phase pipeline with optional quorum review and E2E testing.
</Callout>

## Quick reference

| Field | Value |
|-------|-------|
| Command | `/brewcode:skills` |
| Arguments | `[list\|up\|create] [target]` |
| Model | opus |
| Context | session |
| Tools | Read, Write, Edit, Glob, Grep, Bash, Task, WebSearch, WebFetch, AskUserQuestion |

| Mode | Trigger | Effect |
|------|---------|--------|
| `list` | no args or `list` | Print all skills across global / project / plugins |
| `up` | `up <name\|path\|folder>` or just `<name>` | Improve an existing skill |
| `create` | `create <prompt\|spec-path>` | Build a new skill from scratch |

## When to use

- **Audit your library** — `/brewcode:skills` with no args gives a full table of global, project, and plugin skills
- **Fix a skill that doesn't trigger** — `/brewcode:skills up commit` re-optimizes activation phrases and description
- **Batch-improve a folder** — `/brewcode:skills up &#126;/.claude/skills/` processes every `SKILL.md` in parallel
- **Create from a prompt** — `/brewcode:skills create "semantic code search"` researches and scaffolds a complete skill
- **Create from a spec** — `/brewcode:skills create ./spec.md` reads your spec file as the source of truth

## Examples

```bash
# List all skills (global + project + plugins)
/brewcode:skills

# Improve a skill by name — "up" is the default mode
/brewcode:skills commit

# Improve all skills in a folder at once
/brewcode:skills up &#126;/.claude/skills/

# Create from a text prompt
/brewcode:skills create "semantic code search"

# Create from a prepared spec file
/brewcode:skills create ./spec.md
```

## Flow

Both `create` and `up` run the same 7-phase pipeline. Differences are noted per step.

<Steps>
  <li>
    <div>
      <strong>Discovery</strong>
      <p>2–3 Explore agents run in parallel. For <code>create</code>: scan existing skill patterns, project structure, and optionally web-research the topic. For <code>up</code>: analyze the target skill and compare it against brewcode plugin patterns to identify gaps.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>User interview</strong>
      <p>Seven interactive questions via <code>AskUserQuestion</code>: check conversation history (create only), determine input type (create only), choose invocation type (user / LLM / both), detect Mode Switcher keywords (create only), pick testing depth, pick review type, confirm the plan.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Create / Improve</strong>
      <p>Spawns the <a href="/brewcode/agents/skill-creator/">Skill Creator</a> agent with discovery results and user choices. The agent builds the full skill structure: frontmatter, activation triggers, references, and scripts. Folder targets run parallel agents — one per <code>SKILL.md</code> found.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Validate</strong>
      <p>Automatic inside the Skill Creator agent: structure checks, <code>validate-skill.sh</code>, unit tests for scripts, README generation. No orchestrator action required at this step.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Review</strong>
      <p><strong>Skipped for Quick depth.</strong> Simple review: one reviewer agent plus one verification agent — confirmed findings go back to Skill Creator. Quorum review: three reviewers in parallel, 2/3 threshold, DoubleCheck pass, then Skill Creator fixes verified issues.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>E2E testing</strong>
      <p><strong>Deep depth only.</strong> Creates test scenarios (one per mode + edge cases), executes each in an isolated temp project via <code>claude -p</code>. Failed scenarios retry up to 2 times; major failures loop back to Create/Improve.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Summary</strong>
      <p>Structured report: action, skill path, invocation type, testing depth, review type, phases completed, confirmed findings, test results, and suggestions for follow-up.</p>
    </div>
  </li>
</Steps>

<details>
<summary>Internals</summary>

**Testing depth matrix**

| Depth | Validate | Review | E2E |
|-------|----------|--------|-----|
| Quick (default) | ✅ | — | — |
| Standard | ✅ | Simple (1 reviewer + verify) | — |
| Deep | ✅ | Quorum (3 reviewers, 2/3) | ✅ |

**Mode Switcher pattern**

When `create` detects keywords like "mode", "toggle", "switch", or "persistent", it asks whether to generate a Mode Switcher skill. The generated skill supports three arguments — `on [mode-name]`, `off`, `status` — with state stored in `brewcode.state.json`. Scope priority: session > project > global. A `SessionStart` hook injects mode instructions on every event.

**Folder target resolution**

`up &#126;/.claude/skills/` runs `find <folder> -name "SKILL.md" -type f` and spawns one Skill Creator agent per result — all in a single message for parallel execution.

**Smart mode detection**

If the first argument is not a mode keyword (`list`, `up`, `create`), the entire input is treated as a target for `up`. `/brewcode:skills commit` is identical to `/brewcode:skills up commit`.

</details>
