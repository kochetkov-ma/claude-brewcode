# plugin-update — manage plugin versions

<Callout type="caution">
**Running old plugin versions causes silent mismatches.** Skills added in newer releases are missing. Fixes don't apply. The skill manifest Claude loads at startup can be weeks behind the repo. There is no auto-update prompt — you have to ask.
</Callout>

<Callout type="tip">
**One skill checks, installs, and updates the full brewcode suite.** Interactive by default — or pass `check` / `update` / `all` to skip questions. For individual skill toggling, see [`/brewtools:skill-toggle`](/brewtools/skills/skill-toggle/).
</Callout>

## Quick reference

| Field | Value |
|-------|-------|
| Command | `/brewtools:plugin-update` |
| Arguments | `[check\|update\|all]` — no args = interactive |
| Model | sonnet |
| Context | session |
| Tools | Read, Bash, Write, AskUserQuestion, WebFetch |

## When to use

- **Version check before a long session** — confirm the suite is current before investing time
- **After seeing unfamiliar errors** — an outdated plugin may lack a fix already released
- **First run on a new machine** — install all four plugins non-interactively with `all`
- **CI / scripted setup** — `all` mode skips every confirmation prompt

| Situation | Command |
|-----------|---------|
| Just check versions, change nothing | `/brewtools:plugin-update check` |
| Update what's outdated, skip missing | `/brewtools:plugin-update update` |
| Install missing + update all, no prompts | `/brewtools:plugin-update all` |
| Walk through each decision interactively | `/brewtools:plugin-update` |

## Examples

```bash
# Interactive — check versions and decide what to update
/brewtools:plugin-update

# Status check only — no changes made
/brewtools:plugin-update check
```

```bash
# Update all outdated plugins non-interactively
/brewtools:plugin-update update

# Install missing + update all — fully non-interactive
/brewtools:plugin-update all
```

After a run you will see a version table like this:

| Plugin | Installed | Latest | Status |
|--------|-----------|--------|--------|
| brewcode | 3.4.51 | 3.4.52 | ⬇️ update available |
| brewdoc | 3.4.51 | 3.4.51 | ✅ current |
| brewtools | — | 3.4.51 | ❌ missing |
| brewui | 3.4.51 | 3.4.51 | ✅ current |

Status legend: ✅ current · ⬇️ update available · ❌ missing · ❓ unknown.

## Flow

<Steps>
  <li>
    <div>
      <strong>Discover installed plugins</strong>
      <p>Runs <code>discover-plugins.sh</code>, which reads <code>&#126;/.claude/settings.json</code> and walks the plugin cache at <code>&#126;/.claude/plugins/cache/</code>. Outputs a JSON object with each installed plugin and its version. <code>claude plugin list</code> does not exist as a subcommand — the script bypasses it entirely.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Fetch latest versions</strong>
      <p>Fetches release metadata from GitHub. If the network request fails, all latest versions are marked "unknown" and the run continues — you can still update without knowing the exact latest tag.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Render status table</strong>
      <p>Prints the version table for all four suite plugins (brewcode, brewdoc, brewtools, brewui) plus any other installed plugins as informational rows. Argument <code>check</code> stops here — no changes are made.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Install missing plugins</strong>
      <p>For each suite plugin not present: asks whether to install (interactive) or auto-installs (<code>all</code> mode). Runs <code>claude plugin marketplace add</code> first (idempotent), then per-plugin <code>install</code>. Skipped entirely for the <code>update</code> argument.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Update outdated plugins</strong>
      <p>Runs the full update chain in order: marketplace refresh, then each of the four plugins. Shows full CLI output for every command. In interactive mode, asks before proceeding. Any single failure is reported and the chain continues.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Auto-update toggle hint</strong>
      <p>Only in interactive and <code>all</code> modes. Explains how to enable per-marketplace auto-update via the <code>/plugin</code> UI (Marketplaces → claude-brewcode). The skill never patches <code>settings.json</code> directly — the exact key is unverified.</p>
    </div>
  </li>
  <li>
    <div>
      <strong>Reload notice + final summary</strong>
      <p>Always printed, even on no-op runs. Lists what was installed, updated, skipped, and any errors. Preferred reload: run <code>/reload-plugins</code> in this session. Fallback: <code>exit</code>, then <code>claude</code> again.</p>
    </div>
  </li>
</Steps>

<Callout type="warning">
Run `/reload-plugins` or restart the session after any install or update. Claude Code reads skill and agent definitions once at startup — changes take effect only after a reload.
</Callout>

<details>
<summary>Internals</summary>

**Discovery** — no `claude plugin list`. The skill uses `discover-plugins.sh`, which parses `&#126;/.claude/settings.json` for marketplace registrations and walks `&#126;/.claude/plugins/cache/` for installed version directories.

**Update chain** — runs in this fixed order:

```bash
claude plugin marketplace update claude-brewcode
claude plugin update brewcode@claude-brewcode
claude plugin update brewdoc@claude-brewcode
claude plugin update brewtools@claude-brewcode
claude plugin update brewui@claude-brewcode
```

Each command runs via the Bash tool with full output shown. Failures are reported inline; remaining steps continue regardless.

**Interactive gates** — `AskUserQuestion` presents option lists only (no free-text). Three decision points: install missing, update outdated, enable auto-update. Explicit arguments (`check`, `update`, `all`) bypass all three.

**Claude refuses to execute?** — if Claude summarizes instead of running commands, paste this coercive prompt: _"Execute every `claude plugin ...` command shown above using the Bash tool. Do not describe — run each command and show full output."_
</details>
