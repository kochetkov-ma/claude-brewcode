# Создание скилла `brewcode:e2e` и экосистемы E2E-агентов

> **Контекст:** это начальный промт-спецификация. Перед реализацией необходима стадия планирования — уточнение структуры, frontmatter, моделей, расположения файлов и порядка выполнения. Всё ниже — намерение и направление, не финальное решение. Планирование проводится отдельно.

## Цель

Создать скилл `brewcode:e2e` и набор специализированных E2E-агентов для полного цикла end-to-end тестирования: от анализа проекта и написания сценариев до автоматизации тестов и их ревью. Скилл работает по модели делегирования — менеджер (скилл) управляет агентами через Task tool.

---

## 1. Скилл: `brewcode:e2e`

### Расположение (предварительное, уточнить при планировании)

```

├── SKILL.md
├── README.md
```

### Режимы (modes)

Каждый режим принимает опциональный промт — произвольный текст после имени режима, передающий контекст/инструкции для данного запуска. Если промт не указан — скилл уточняет через AskUserQuestion. Если указан — использует как начальный контекст, но всё равно подтверждает план через AskUserQuestion.

| Режим | Вызов | Описание |
|-------|-------|----------|
| `setup` (default) | `/brewcode:e2e [prompt]` или `/brewcode:e2e setup [prompt]` | Анализ проекта → создание E2E-агентов → генерация правил → фиксация документации |
| `create` | `/brewcode:e2e create [prompt]` | Создание E2E-сценариев и тестов по описанию flow/части системы |
| `update` | `/brewcode:e2e update [prompt]` | Улучшение существующих сценариев/тестов |
| `review` | `/brewcode:e2e review [prompt]` | Мультиагентное ревью с quorum (3 ревьюера, консенсус 2/3) |
| `rules` | `/brewcode:e2e rules [prompt]` | Создание/обновление/улучшение правил E2E-тестирования |
| `status` | `/brewcode:e2e status [prompt]` | Проверка настроек: агенты, правила, сценарии — вывод статуса |

### Общие принципы скилла

1. **AskUserQuestion всегда.** На каждом ключевом этапе — уточнять у пользователя. Промт пользователя — не замена подтверждения, а начальный контекст. Если сценарий уже есть — спросить по какому работаем. Если нет — предложить создать.
2. **Трёхшаговый цикл: Выполнение → Валидация → Перепроверка.** Каждое действие проходит цикл: агент делает → другой агент валидирует → найденные проблемы перепроверяются → исправляются. Цикл повторяется до полного исправления.
3. **Без сценария — не пишем тесты.** E2E-тесты создаются ИСКЛЮЧИТЕЛЬНО по заранее написанному и проверенному сценарию. Сценарий должен быть воспроизводим вручную.
4. **Делегирование через агентов.** Скилл — менеджер, вся работа через Task tool.

## 2. Агенты E2E

### Список агентов (предварительный)

Размещение, модели и точный набор инструментов — определить при планировании.

| Агент | Роль |
|-------|------|
| `e2e-manager` | QA-менеджер: координация E2E-агентов, распределение задач, контроль качества |
| `e2e-manual-tester` | Ручной тестировщик: проверяет систему через UI/API, находит баги, работает с credentials |
| `e2e-automation-tester` | Автоматизатор: пишет E2E-автотесты по готовым сценариям, строго по архитектурным правилам |
| `e2e-scenario-analyst` | Сценарный аналитик: создаёт детальные E2E-сценарии в BDD-формате на основе анализа системы |
| `e2e-reviewer` | Ревьюер: проверяет качество тестов и сценариев, соответствие правилам, полноту покрытия (READ-ONLY) |
| `e2e-architect` | Архитектор E2E: анализирует проект, определяет паттерны для тестов, создаёт правила и конвенции |

> **Открытые вопросы по агентам:**
> - Нужны ли дополнительные агенты? (e2e-data-engineer, e2e-infra — скилл может предложить после анализа проекта)
> - Модели для каждого агента — определить при планировании
> - Точный набор tools для каждого — определить при планировании
> - disallowedTools для read-only агентов — уточнить

### Шаблон агентов

Взять шаблон из `brewcode/skills/teams/references/agent-template.md` — скопировать и адаптировать под E2E-контекст. НЕ ссылаться на оригинал — именно копия с модификациями.

Обязательные элементы шаблона (из teams):
- Mission / Domain / Character / Last Updated
- Immutable Traits (Name, Base Role)
- Task Acceptance Protocol (Domain / Duplicate / Best candidate проверки)
- Domain Instructions (специфичные для E2E)
- Colleagues (таблица коллег из E2E-команды)

Дополнительные требования к агентам:
- **Самокритичность:** каждый агент после выполнения задачи проверяет свой результат перед отдачей менеджеру
- **Загрузка правил:** агенты, которые пишут код или ревьюят, ОБЯЗАНЫ загружать правила E2E перед началом работы. Если файл не найден — агент останавливается и сообщает менеджеру
- **Lazy vs Eager:** по умолчанию eager (загружать и валидировать ДО начала работы). Если в инструкциях явно указано `lazy` — загружать по требованию
- **Создание через agent-creator:** все агенты создаются через `Task(subagent_type="brewcode:agent-creator")` батчами по 3-4

## 3. Правила E2E-тестирования (references/e2e-rules.md)

### Задача по правилам

**При setup:** провести веб-исследование лучших практик E2E-тестирования (WebSearch), собрать и структурировать полный набор правил. Правила — короткие, конкретные, actionable.

**При режиме `rules`:** обновить/улучшить правила на основе опыта текущего проекта. Перепроверять все источники при обновлении.

### Базовый набор правил (минимум, расширить при исследовании)

#### Сценарии

| # | Правило | Пояснение |
|---|---------|-----------|
| S1 | Без сценария — не пишем тест | E2E-тест создаётся ИСКЛЮЧИТЕЛЬНО по утверждённому сценарию |
| S2 | Сценарий = полный пользовательский flow | От начала до конца, с осязаемой ценностью для пользователя. Не фрагмент, а законченный бизнес-путь |
| S3 | Сценарий воспроизводим вручную | Каждый шаг можно выполнить руками (включая API-запросы через curl/Postman) |
| S4 | BDD-формат: Given/When/Then | С уточнением после двоеточия: `Given: пользователь авторизован как admin` |
| S5 | Сценарии могут пересекаться | Авторизация — общий шаг в разных сценариях. Это нормально для E2E |
| S6 | Тесты длинные | E2E — это полные flow. Один тест проходит через множество шагов и систем |

#### Тестовые данные

| # | Правило | Пояснение |
|---|---------|-----------|
| D1 | Максимальный набор тестовых данных | Не минимальный «2 записи», а покрытие основных кейсов с запасом |
| D2 | Данные через API, не напрямую в БД | Через публичный API — идеал. Через UI — допустимо. Прямая запись в БД — крайний случай с подтверждением пользователя |
| D3 | Отдельный слой генерации/хранения данных | Data layer как сервис, вызываемый из тестов |
| D4 | Параметризованные тесты обязательны | Один тест — множество наборов данных |

#### Интеграция

| # | Правило | Пояснение |
|---|---------|-----------|
| I1 | Полная интеграция, без моков | Моки — только если интеграция технически невозможна, с подтверждением пользователя |
| I2 | Проверка во ВСЕХ конечных системах | Проверять не только основную систему, но и все downstream-зависимости |
| I3 | Тесты проверяют пользовательский flow | Не техническую реализацию, а бизнес-результат |

#### Ассерты

| # | Правило | Пояснение |
|---|---------|-----------|
| A1 | Строгие ассерты: сравнение объектов | Никаких `isNotNull()`, `isNotEmpty()`, `isGreaterThanOrEqualTo(0)` |
| A2 | Конкретные значения | `isEqualTo(expectedValue)`, `hasSize(N)`, полное сравнение объектов |
| A3 | Описание на каждом ассерте | `.as("описание проверки")` или эквивалент фреймворка |
| A4 | Нет `if` в тестах | Никогда `if (size > 1) { assert... }`. Assert precondition сначала, потом безусловная проверка |

#### Архитектура (краткие правила, подробнее в e2e-architecture.md)

| # | Правило | Пояснение |
|---|---------|-----------|
| R1 | Слой шагов (Steps) отделяет бизнес от техники | В тестах — бизнес-шаги, без прямых вызовов API/UI-элементов |
| R2 | Базовые классы по доменам | `BaseE2E` → `BaseAuthE2E`, `BasePaymentE2E` и т.д. |
| R3 | Support-классы вместо Utils | `KafkaSupport`, `DatabaseSupport` — не пересекаться с библиотечными утилитами |
| R4 | Репортер вместо логов | Reporter тестового фреймворка. Никаких `print`/`System.out` |

#### Процесс

| # | Правило | Пояснение |
|---|---------|-----------|
| P1 | Трёхшаговый цикл | Выполнение → Валидация (другой агент) → Перепроверка → Исправление |
| P2 | Перепроверка КАЖДОЙ проблемы после ревью | Ревьюер может ошибиться. Убедиться, что проблема реальна, перед исправлением |
| P3 | Правила — в документации проекта | Фиксируются рядом с агентами и доступны для загрузки |

## 4. Архитектура E2E-тестов (references/e2e-architecture.md)

Целевая архитектура для `e2e-architect` и `e2e-automation-tester`:

```
┌─────────────────────────────────────────────┐
│  Test Classes (по доменам)                  │
│  extends BaseE2E / BaseDomainE2E            │
│  содержат: @ParameterizedTest + шаги        │
├─────────────────────────────────────────────┤
│  Steps Layer (бизнес-шаги)                  │
│  переиспользуемые across tests              │
│  Given/When/Then шаги                       │
├─────────────────────────────────────────────┤
│  Verification Layer (assertion steps)       │
│  строгие проверки, полное сравнение         │
├─────────────────────────────────────────────┤
│  Data Layer                                 │
│  генерация / хранение / подготовка данных   │
│  через API (не прямая запись в БД)          │
├─────────────────────────────────────────────┤
│  Support Layer                              │
│  KafkaSupport, DatabaseSupport, HttpSupport │
│  технические утилиты интеграции             │
├─────────────────────────────────────────────┤
│  Config Layer                               │
│  настройки тестового окружения              │
│  credentials, endpoints, timeouts           │
└─────────────────────────────────────────────┘
```

### Ключевые архитектурные требования

| # | Требование | Детали |
|---|-----------|--------|
| 1 | В тестах нет прямого кода | Нет прямых вызовов API, нет прямых обращений к UI-элементам — всё через Steps |
| 2 | Steps = бизнес-язык | `givenUserIsAuthorized("admin")`, не `httpClient.post("/auth", ...)` |
| 3 | Параметризованные тесты | Один тестовый метод — множество наборов данных |
| 4 | Базовые классы | `BaseE2E` (общий) → `Base{Domain}E2E` (по доменам) → конкретные тесты |
| 5 | Support вместо Utils | Именование: `{Technology}Support` (не `{Technology}Utils`) |
| 6 | Given/When/Then разделение | Через возможности фреймворка если есть, иначе через комментарии |
| 7 | Аннотации и метаданные | Максимально использовать возможности фреймворка: `@DisplayName`, `@Tag`, `@Step` и т.д. |
| 8 | Репортинг через фреймворк | Reporter > Logger > ничего |

> `e2e-architect` анализирует конкретный проект и адаптирует архитектуру под стек и conventions проекта. Этот документ — отправная точка, не догма.

## 5. Детализация режимов

### 5.1 setup (default)

**Цель:** подготовить проект к E2E-тестированию — создать агентов, правила, документацию.
**Промт:** уточняет scope настройки. Примеры: `"focus on payment domain"`, `"only API tests, no UI"`, `"use Playwright + Jest"`.

Примерная последовательность (уточнить при планировании):
1. Проверка статуса: если уже настроено — вывести и предложить другой режим
2. Анализ проекта (параллельно 3-5 Explore-агентов): структура, тесты, стек, API, инфраструктура
3. AskUserQuestion: результат анализа, уточнение scope, подтверждение плана
4. Создание E2E-агентов через agent-creator (батчами)
5. Анализ архитектуры + WebSearch лучших практик → генерация правил
6. Валидация правил (e2e-reviewer)
7. Фиксация документации, AskUserQuestion: итог
8. Опционально: вынести краткие правила в `.claude/rules/`

### 5.2 create

**Цель:** создать E2E-сценарии и тесты.
**Промт:** описывает что тестируем. Примеры: `"checkout flow with 3D Secure"`, `"user registration and email verification"`.

Примерная последовательность:
1. AskUserQuestion: уточнить scope (промт как стартовый контекст)
2. Анализ целевой части системы (3-5 Explore параллельно)
3. Создание сценариев (e2e-scenario-analyst)
4. Валидация сценариев (e2e-reviewer) → перепроверка → исправление
5. AskUserQuestion: показать сценарии, получить одобрение
6. Написание E2E-тестов по сценариям (e2e-automation-tester)
7. Ревью тестов (e2e-reviewer) → перепроверка → исправление (цикл до 0 проблем)
8. AskUserQuestion: итоговый результат

### 5.3 update

**Цель:** улучшить существующие сценарии/тесты.
**Промт:** описывает что улучшить. Примеры: `"add negative scenarios to payment tests"`, `"refactor auth steps to use new API"`.

Аналогично create, но начинается с поиска существующих артефактов и вопроса что улучшить.

### 5.4 review

**Цель:** мультиагентное ревью с quorum.
**Промт:** ограничивает scope. Примеры: `"review only auth e2e tests"`, `"focus on assertion quality"`.

Примерная последовательность:
1. AskUserQuestion: что ревьюим (промт как стартовый scope)
2. Разделить scope на части
3. На каждую часть — 3 параллельных e2e-reviewer (quorum 2/3)
4. Ревью сценариев: проверка соответствия сценариев и тестов
5. Перепроверка. Консенсус: проблема подтверждена если 2 из 3 согласны
6. AskUserQuestion: результат, предложение исправлений

### 5.5 rules

**Цель:** создание / обновление / улучшение правил E2E.
**Промт:** фокус обновления. Примеры: `"add rules for async messaging tests"`, `"research Playwright best practices"`.

Примерная последовательность:
1. Загрузить текущие правила, проверить актуальность
2. WebSearch: актуальные практики для стека проекта (с учётом промта)
3. e2e-architect: анализ проекта, поиск закономерностей
4. Обновление правил с перепроверкой всех источников
5. AskUserQuestion: diff правил, подтверждение
6. Опционально: вынести summary в `.claude/rules/` по запросу пользователя

При обновлении правил — все источники перепроверять. Правила короткие, конкретные, actionable.

### 5.6 status

**Цель:** проверка текущего состояния E2E-инфраструктуры.
**Промт:** фильтрует вывод. Примеры: `"show only missing agents"`, `"check rules freshness"`.

| Проверка | Что смотрим |
|----------|-------------|
| Агенты | Существуют ли `e2e-*` в `.claude/agents/`? |
| Правила | Файл правил E2E? Количество? |
| Сценарии | Написанные сценарии? Расположение? |
| Тесты | E2E-тесты? Структура каталогов? |
| Конфигурация | Стек, фреймворк, базовые классы |

## 6. Технические ориентиры

> Это направляющие принципы, а не финальные решения. Конкретные детали — при планировании.

### Создание агентов
- Через `Task(subagent_type="brewcode:agent-creator")` — НЕ вручную
- Батчами по 3-4 агента в одном сообщении (параллельно)
- После создания — оптимизация через `Skill(skill="brewtools:text-optimize")`
- Шаблон — адаптация из teams agent-template

### Создание скилла
- Через `Task(subagent_type="brewcode:skill-creator")`
- Паттерны: Progressive Disclosure, Reference Splitting, Mode Switcher (detect-mode.sh)
- `${CLAUDE_SKILL_DIR}` в SKILL.md, `$BC_PLUGIN_ROOT` в промтах агентов
- Маркер `**EXECUTE** using Bash tool:` + `&& echo "OK" || echo "FAILED"`

### Правила и документация
- Основные правила — в `references/e2e-rules.md` (полный набор для агентов)
- Архитектура — в `references/e2e-architecture.md`
- Опционально: краткие выжимки в `.claude/rules/e2e-avoid.md` и `e2e-best-practice.md` (по запросу пользователя)
- При вынесении в `.claude/rules/` — дедупликация обязательна

### Валидация
- Каждый артефакт (агент, скилл, правило) валидируется после создания
- Финальное ревью всего результата
- Все находки ревью перепроверяются перед исправлением
